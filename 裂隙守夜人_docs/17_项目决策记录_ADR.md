# 项目架构决策记录 ADR

## ADR-001：采用 Godot 4.x

原因：

- 适合2D
- GDScript开发效率高
- 场景和资源体系适合数据驱动
- 方便个人开发

## ADR-002：采用横版竞技场而非线性关卡

原因：

- 更适合持续涌兵
- 控制地图制作成本
- 方便重复游玩
- 便于移动端适配

## ADR-003：采用主动攻击而非自动攻击

原因：

- 强调动作手感
- 与普通割草游戏区分
- 让闪避、连击和技能释放具有价值

## ADR-004：技能使用 Resource 数据驱动

原因：

- 后续技能数量较多
- 便于平衡数值
- 降低 Player 脚本复杂度

## ADR-005：敌人和常用特效使用对象池

原因：

- 敌人持续生成
- 高频创建销毁会造成卡顿
- 未来移动端更需要控制性能

## ADR-006：MVP 不做装备随机词条

原因：

- 会显著扩大数值和 UI 复杂度
- 核心乐趣尚未验证
- 与技能构筑存在系统重叠

## ADR-007：首版只做一名角色和一张地图

原因：

- 优先验证战斗和成长
- 避免美术成本过早膨胀
- 保证可完成性
