# 技术架构设计

## 1. 架构目标

- 模块化
- 数据驱动
- 易于新增技能和敌人
- 避免节点之间强耦合
- 支持对象池
- 支持存档
- 支持未来移动端输入

## 3. Autoload

建议只保留少量全局单例：

### GameManager

职责：

- 游戏状态
- 场景切换
- 暂停
- 单局开始和结束

### EventBus

职责：

- 公开跨系统信号
- 降低直接引用

### SaveManager

职责：

- 设置存档
- 局外进度
- 存档版本迁移

### AudioManager

职责：

- 播放全局音效
- 音频总线控制
- 音效并发限制

### ObjectPoolManager

职责：

- 投射物
- 伤害数字
- 普通敌人
- 常用特效

禁止把玩家、敌人或 UI 具体逻辑放入 Autoload。

## 5. 组件化

通用组件：

- HealthComponent
- HurtboxComponent
- HitboxComponent
- StatusEffectComponent
- MovementComponent
- KnockbackComponent
- SkillComponent
- CooldownComponent
- LootDropComponent
- FlashComponent

组件负责单一职责。

## 6. 数据 Resource

建议资源类型：

- PlayerData
- EnemyData
- SkillData
- SkillUpgradeData
- PassiveUpgradeData
- WaveData
- BossPhaseData
- BalanceData

Resource 保存静态配置。

运行时状态不得直接写回共享 Resource，必要时复制或单独保存实例状态。

## 7. 信号设计

推荐信号：

- player_health_changed
- player_died
- enemy_spawned
- enemy_died
- experience_gained
- player_level_up
- upgrade_selected
- skill_cast
- skill_cooldown_changed
- ultimate_energy_changed
- boss_spawned
- boss_phase_changed
- battle_finished

信号参数必须清晰，避免传递无意义字典。

## 8. 战斗上下文

伤害事件使用 DamageContext：

- source
- target
- base_damage
- final_damage
- damage_type
- tags
- is_critical
- knockback
- hit_position
- status_effects

这样便于后续增加暴击、元素、伤害统计和被动触发。

## 9. 对象池

必须优先池化：

- 敌人
- 弹幕
- 剑气
- 命中特效
- 伤害数字
- 经验球

池对象需要实现：

- activate
- deactivate
- reset_state

回收时必须：

- 停止计时器
- 停止粒子
- 清空信号连接或避免重复连接
- 重置速度
- 重置生命值
- 禁用碰撞

## 10. 存档

JSON 存档结构：

```json
{
  "version": 1,
  "settings": {},
  "meta_progress": {},
  "unlocks": {},
  "statistics": {}
}
```

保存内容：

- 解锁技能
- 解锁武器
- 天赋
- 货币
- 最高纪录
- 设置

不保存正在进行的单局。

## 11. 错误处理

- 资源加载失败时输出明确错误
- 缺失图标时使用默认占位图标
- 缺失音效时不中断游戏
- 无效技能配置禁止进入升级池
- 无效敌人配置跳过生成并记录错误

## 12. 编码规范

- 文件名：snake_case
- 类名：PascalCase
- 变量和函数：snake_case
- 信号：snake_case
- 常量：UPPER_SNAKE_CASE
- 每个脚本尽量不超过400行
- 超过后拆分组件
- 公开方法写注释
- 避免深层节点路径硬编码

## 13. 验收标准

- 新增一个敌人主要通过创建场景和 Resource 完成
- 新增一个技能不需要修改玩家主控制器
- UI 不直接处理战斗计算
- WaveManager 不直接写敌人内部状态
- 所有系统可以独立测试
